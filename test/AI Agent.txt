AI Agent Session – Autonomous Invoice Processing Automation

Goal
Build an automation workflow where invoices move through multiple processing stages automatically using a state-driven orchestration system. The objective is to allow the system to ingest invoices, validate them, match them with purchase orders, route approvals, and schedule payments with minimal human intervention.

This session documents how AI was used to generate the initial orchestration logic and how it was corrected and improved.

--------------------------------------------------

Step 1 – Prompt Given to AI

Prompt:

I am building an AI-driven finance automation system in Node.js.

Design an orchestration logic where invoices move through states such as:

RECEIVED
STRUCTURED
DUPLICATE_CHECK
VALIDATING
MATCHING
EXCEPTION_REVIEW
PENDING_APPROVAL
APPROVED
PAYMENT_READY
COMPLETED

The orchestrator should:
1. Read the current invoice state from the database
2. Trigger the correct worker based on the state
3. Update the state after the worker completes
4. Continue processing until the invoice reaches a terminal state

Use a switch-based state machine design.

--------------------------------------------------

Step 2 – Code Generated by AI (Initial Version)

async function processInvoice(invoice) {

  if (invoice.state === "RECEIVED") {
    await intakeWorker(invoice);
    invoice.state = "STRUCTURED";
  }

  if (invoice.state === "STRUCTURED") {
    await validationWorker(invoice);
    invoice.state = "VALIDATING";
  }

  if (invoice.state === "VALIDATING") {
    await matchingWorker(invoice);
    invoice.state = "MATCHING";
  }

  if (invoice.state === "MATCHING") {
    await approvalWorker(invoice);
    invoice.state = "PENDING_APPROVAL";
  }

}

Problems identified:

1. Linear execution – the system only runs once and does not re-evaluate state changes.
2. No persistent state reads from database.
3. No handling for exception states or manual review.
4. No loop to allow continuous orchestration.

--------------------------------------------------

Step 3 – Fixes Applied

To solve the issues, the orchestration logic was redesigned as a state-driven processing loop.

Key changes:

- Introduced a loop that continuously reads invoice state.
- Used a switch-based state machine.
- Added workers for each stage of processing.
- Ensured state transitions are written back to the database.
- Allowed exception handling and manual intervention states.

--------------------------------------------------

Step 4 – Final Automation Logic

async function orchestrateInvoice(invoiceId) {

  let invoice = await getInvoice(invoiceId);

  while (invoice.state !== "COMPLETED" && invoice.state !== "BLOCKED") {

    switch (invoice.state) {

      case "RECEIVED":
        await intakeExtractionWorker(invoice);
        await updateState(invoiceId, "STRUCTURED");
        break;

      case "STRUCTURED":
        await duplicateCheckWorker(invoice);
        await updateState(invoiceId, "DUPLICATE_CHECK");
        break;

      case "DUPLICATE_CHECK":
        await validationWorker(invoice);
        await updateState(invoiceId, "VALIDATING");
        break;

      case "VALIDATING":
        await validationAgent(invoice);
        await updateState(invoiceId, "MATCHING");
        break;

      case "MATCHING":
        await matchingWorker(invoice);
        await updateState(invoiceId, "PENDING_APPROVAL");
        break;

      case "PENDING_APPROVAL":
        await approvalAgent(invoice);
        await updateState(invoiceId, "APPROVED");
        break;

      case "APPROVED":
        await paymentWorker(invoice);
        await updateState(invoiceId, "PAYMENT_READY");
        break;

      case "PAYMENT_READY":
        await executePayment(invoice);
        await updateState(invoiceId, "COMPLETED");
        break;

      case "EXCEPTION_REVIEW":
        await exceptionReviewAgent(invoice);
        break;

      case "WAITING_INFO":
        break;

      default:
        console.log("Unknown invoice state");
        break;

    }

    invoice = await getInvoice(invoiceId);

  }

}

--------------------------------------------------

Step 5 – Architecture Summary

The system follows a state-driven automation architecture.

1. An invoice enters the system with state = RECEIVED.
2. The orchestrator reads the current state.
3. The corresponding worker or agent is triggered.
4. The worker completes its task and updates the state.
5. The orchestrator re-checks the state and continues processing.

Workers perform deterministic tasks such as extraction, validation, and payment scheduling.

Agents perform reasoning tasks such as approval decisions, risk evaluation, and exception handling.

The orchestrator ensures the system progresses through the workflow until the invoice reaches a terminal state.

This design allows the system to operate autonomously while maintaining auditability, control, and recoverability.
